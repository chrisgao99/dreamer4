# Waymo Vector Data Quality Audit

Last updated: 2026-05-19

## Purpose

This note records the current plan and tooling for understanding the filtered
Waymo vector dataset used by the Dreamer-4-style tokenizer.

The immediate goal is to answer:

- What kinds of scenarios are in the 5k filtered dataset?
- How many are straight, stopped/slow, lane-change-like, turning, or
  intersection-like?
- How many have close ego-agent interactions, following/leading behavior,
  crossing-like interactions, or cut-in/merge-like cues?
- Are the filtered agent, map, and traffic-light tensors internally sane?

The labels produced by the audit are heuristic labels inferred from the vector
tensors. They are not official Waymo semantic labels. They should be used to
guide visual inspection, dataset balancing, and downstream experiment design.

## Dataset Being Audited

Current 5k filtered dataset:

```text
/p/yufeng/tri30/dreamer4/data/waymo_vector_dataset_5k
```

This directory contains 5000 `.npz` files. Each file is one Waymo Motion
scenario filtered into ego-centric vector tensors:

```text
agents:        (16, 91, 8)
agent_mask:    (16,)
map_polylines: (256, 20, 6)
map_mask:      (256, 20)
lights:        (91, 16, 4)
light_mask:    (91, 16)
```

The 91 timesteps correspond to:

```text
10 past + 1 current + 80 future
```

At 10 Hz, this is about 9.1 seconds of trajectory data per scenario.

The 5k dataset was generated by:

```text
/p/yufeng/tri30/dreamer4/waymo/prepare_waymo_vector_dataset.sh
```

Generation settings:

```text
RAW_DIR=/p/liverobotics/waymo_open_dataset_motion/tf_example/training
N_FILES=50
MAX_RECORDS=100
NUM_AGENTS=16
AGENT_DISTANCE_THRESHOLD=80
MAP_DISTANCE_THRESHOLD=100
MAX_MAP_POLYLINES=256
MAX_POINTS_PER_POLYLINE=20
--history_only_selection
```

This means the 5k split is the first 100 scenarios from each of the first 50
training TFRecord files.

## Ego Maneuver Labels

The first-pass ego maneuver labels are:

```text
stopped_or_slow
straight
lane_change_like
left_turn
right_turn
large_turn_or_uturn
curved_or_other_moving
```

Current thresholds:

```text
stopped_or_slow:
  ego displacement < 2m OR ego path length < 3m

straight:
  abs heading change <= 15 degrees
  abs lateral displacement <= 2m

lane_change_like:
  abs heading change <= 30 degrees
  abs lateral displacement >= 2.5m
  longitudinal displacement >= 10m

left_turn/right_turn:
  abs heading change >= 45 degrees

large_turn_or_uturn:
  abs heading change >= 120 degrees
```

These labels use the full 91-step scenario, not only the observed 11-step
history. They answer "what happened in this scenario" rather than "what was
known at current time."

## Interaction Labels

The audit adds semicolon-separated interaction labels:

```text
close_agent_10m
leading_agent_current
following_agent_current
crossing_conflict_like
cut_in_or_merge_like
dense_scene
sparse_scene
intersection_like
```

Current definitions:

```text
close_agent_10m:
  any non-ego agent comes within 10m of ego at any valid timestep

leading_agent_current:
  current-frame same-direction agent within 30m ahead and within 4m lateral

following_agent_current:
  current-frame same-direction agent within 30m behind and within 4m lateral

crossing_conflict_like:
  close agent within 15m and heading difference >= 45 degrees

cut_in_or_merge_like:
  non-ego agent lateral displacement >= 2.5m, min ego distance <= 20m,
  and positive longitudinal displacement > 5m

dense_scene:
  valid agents at current timestep >= 10

sparse_scene:
  valid agents at current timestep <= 3

intersection_like:
  traffic-light signal exists with light_valid_ratio >= 0.05,
  OR a crossing-like close agent exists,
  OR ego maneuver is a turn/large turn
```

These interaction labels are intentionally broad. They are meant to retrieve
candidate examples for visual review.

## Quality Flags

Possible quality flags include:

```text
missing_<key>
bad_shape_<key>_<shape>
nonfinite_<key>
ego_current_invalid
ego_low_valid_steps
ego_current_not_at_origin
ego_current_yaw_not_zero
no_other_valid_agents
agent_invalid_values_nonzero
map_empty
map_saturated
map_invalid_values_nonzero
no_valid_traffic_lights
light_invalid_values_nonzero
```

Important interpretation:

- `map_saturated` means all 256 map polyline slots are filled. This is not
  necessarily invalid, but it suggests the map crop is often capped by
  `MAX_MAP_POLYLINES=256`.
- `no_valid_traffic_lights` is not necessarily invalid. Highway scenes may have
  no lights.
- `ego_current_not_at_origin` or `ego_current_yaw_not_zero` would be more
  concerning because the filter is supposed to normalize into the ego current
  frame.

## Recommended Manual Review Workflow

After running the full audit:

1. Read `vector_dataset_quality_summary.json` for high-level counts.
2. Open `vector_dataset_scene_labels.csv` and sample scenario ids from each
   `ego_maneuver_label`.
3. Generate preview images/videos for samples from:
   - straight
   - stopped_or_slow
   - lane_change_like
   - left_turn
   - right_turn
   - crossing_conflict_like
   - cut_in_or_merge_like
   - scenarios with `map_saturated`
4. Use visual inspection to tune thresholds before treating labels as dataset
   statistics.

Visualization command example:

```bash
cd /p/yufeng/tri30/dreamer4
/p/yufeng/.conda/envs/dreamer4/bin/python waymo/visualize_waymo_vector_npz.py \

  --output /p/yufeng/tri30/dreamer4/waymo/reports/vector_dataset_5k_audit/previews/<scenario_id>.mp4 \
  --preview_frames 20 \
  --preview_cols 5
```

## Open Questions

- Are `lane_change_like` and `cut_in_or_merge_like` too broad because they use
  simple lateral displacement in the ego-current coordinate frame?
- Is `intersection_like` too broad when any traffic-light validity qualifies?
- Should map quality be re-run with `MAX_MAP_POLYLINES=512` or 768 to measure
  how much information is lost by the current 256 cap?
- Should future audits separate labels based on observed 11-step history from
  labels based on the full 91-step outcome?
- Should scenario labels eventually be enriched from raw Waymo roadgraph/lane
  topology or from the existing custom JSON format?

Important caveat from the first run:

- The interaction labels are currently high-recall and broad. For example,
  `crossing_conflict_like` and `cut_in_or_merge_like` should be treated as
  candidate retrieval labels that need visual inspection, not final semantic
  classes.
- `intersection_like` is also broad because it fires on traffic-light presence,
  turning, or crossing-like interactions. It should be refined after visual
  review.
- `map_saturated` is extremely common, suggesting that `MAX_MAP_POLYLINES=256`
  often caps the local map crop. This should be tested with 512 or 768 map
  polylines before relying on map reconstruction or map-heavy representation
  conclusions.
